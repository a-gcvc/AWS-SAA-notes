- **Identity Policies** are attached to AWS identities and either ALLOW or DENY access to AWS resources.

- Grant access or denies access to AWS products and features to any identity which uses that policy.

- Created using JSON.

## Statements

- **Statement ID, SID** is optional field which lets you identify a statement and what it does.
A statement only applies if the interaction that you're having with AWS match the action and the resource.

**Every interaction that you have with AWS is a combination of two main things, the resource that you're interacting with, and the actions that you're attempting to perform on that resource.**

- **Action** part of a statement matches one or more actions. 

- **Resource**

- **Effect**: allow or deny; effect controls what AWS does if the action and the resource parts of the statement match the operation that you're attempting to do with AWS.

- With the exception of the Account Root User, AWS identites start off with no access to any AWS resources.

- **Two types of policies**:

1. **Inline policies**: design policy and grant access to the resources needed for that project and then assign that JSON individually on every account seperate.
Use when you need to ensure that a specific set of rights, either giving an individual more access rights or blocking certain permissions for one individual indentity, for special or exceptional allows or denies.

2. **Managed policies** are created as their own object; first, you create the managed policy and then you attach that policy to any identites who you wanted to gain access rights.
Should be used for default operational rights in your business.
Two types:
    - AWS managed policies
    - custom managed policies
