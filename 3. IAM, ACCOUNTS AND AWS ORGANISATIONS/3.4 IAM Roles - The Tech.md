- A role is generally best suited to be used by an unkonwn number of multiple principals, not just the one.

- If you can't identify the number of principals which use an identity, then it could be a candidate for an IAM role.

- If you have more then 5000 principals, because of the limit of IAM user it could also be a candidate for an IAM role.

- Roles are generally used on a temporary basis.

- Role is something that no represents you, it is a **level of access inside an AWS account.**

It's a thing that can be used, short term, by other identities.

- With a role, you essentially borrow the permissions for a short period of time.

![alt text](<Screenshot 2025-08-17 135635.png>)

- IAM roles have **two** types of policies which can be attached:

1. **trust policy** controls which identites can assume that role; it can reference identities in the same account, so other IAM users, other roles, and even AWS services such as EC2, and a trust policy can also reference identities in other AWS accounts.

2. **permissions policy**

If a role gets assumed by something which is allowed to assume it, then AWS generates **temporary security credentials** and these are made available to the identity which assumed the role.

temporary security credentials are time-limited and are able to access whateever AWS resources are specified within the permission's policy.

- Roles can be referenced within resource policies.

- When you assume role, temporary credentials are generated by AWS service called STS, Secure Token Service, and this is the operation that's used to assume the role and get the credentials.
